## Executive Summary

This pipeline implements a **multimodal Teacher-Student knowledge distillation** framework (Learning Using Privileged Information — LUPI) for non-destructive mango shelf-life estimation, fusing EfficientNet-B0 visual features with ambient sensor data. The code architecture is clean, modular, and well-documented — a strong foundation for a final year project.

However, the pipeline contains **several critical methodological weaknesses** that, if unresolved, would significantly undermine the academic credibility of the work during viva voce examination or thesis defense. This document identifies each issue, explains its academic consequences, provides a severity rating, and offers concrete implementation guidance for resolution.

> [!IMPORTANT]
> **Overall Assessment**: The project demonstrates strong software engineering and a sophisticated understanding of multimodal deep learning. However, the experimental methodology requires significant strengthening before it can be presented as a rigorous final year project. The issues identified below are ordered by severity.

## 1. CRITICAL — Data Leakage in Train/Validation Split

### Severity: 🔴 Critical (Would result in major mark deductions)

### The Problem

In [`train.py` (lines 201–206)](file:///c:/Users/julth/repo/Smart-Shelf-Life-Prediction-Model/model/src/train.py#L201-L206), the dataset is split using sequential index slicing:

```python
train_df = df.iloc[:train_size].reset_index(drop=True)
val_df   = df.iloc[train_size:].reset_index(drop=True)
```

Examination of [`preprocessed_data.csv`](file:///c:/Users/julth/repo/Smart-Shelf-Life-Prediction-Model/model/preprocessed_data.csv) reveals that every JSON metadata file generates **2 images** (e.g., `aad1img1d1.png` and `aad1img2d1.png`) and **3 samples** share identical biochemical metadata per day (e.g., `aad1`, `aad2`, `aad3` on Day 1). This means:

-   **6 rows per environment-protocol-day combination** share the **same biochemical target values** and nearly identical visual characteristics (solid-colour dummy images with only ±15 RGB noise).
-   The sequential split places some of these near-identical rows in training and others in validation.

### Academic Consequence

The model is effectively evaluated on data it has already seen. Reported validation metrics (MAE ≈ 2.22 days) are **artificially optimistic** and do not reflect generalisation to unseen fruit. Any peer reviewer or thesis examiner familiar with machine learning methodology would identify this immediately.

### Resolution: Group-Based Splitting

Split by **unique Sample ID** (e.g., `aad1`, `cbd3`) to ensure a fruit's entire degradation history stays entirely within either the training or validation set:

```python
import re
from sklearn.model_selection import GroupKFold

def extract_sample_id(image_path: str) -> str:
    filename = image_path.replace('\\', '/').split('/')[-1]
    match = re.match(r'^([a-zA-Z]+\d+)', filename)
    return match.group(1) if match else 'unknown'

df['Sample_ID'] = df['Image Path'].apply(extract_sample_id)

gkf = GroupKFold(n_splits=5)
for fold, (train_idx, val_idx) in enumerate(gkf.split(df, groups=df['Sample_ID'])):
    train_df = df.iloc[train_idx].reset_index(drop=True)
    val_df   = df.iloc[val_idx].reset_index(drop=True)
    break  # Use first fold
```

## 2. CRITICAL — Training Exclusively on Synthetic Data

### Severity: 🔴 Critical (Undermines scientific validity)

### The Problem

The entire training dataset is generated by [`generate_dummy_data.py`](file:///c:/Users/julth/repo/Smart-Shelf-Life-Prediction-Model/model/generate_dummy_data.py). Examination of the generator (lines 92–118) reveals that every "mango image" is a **solid-colour 224×224 PNG** with a deterministic green→yellow→brown gradient plus ±15 RGB noise:

```python
# Line 98: progress = min(day / max_days, 1.0)
# Colour is purely a function of day/max_days
```

This means:
1.  **The image branch is learning to map uniform RGB values to a day-index** — not extracting real visual features (bruising, spots, texture gradients, skin creasing) that a real mango exhibits.
2.  The **tabular measurements** (Brix, pH, Texture) are generated via smooth linear interpolation with small Gaussian noise. Real-world biochemical trajectories are non-linear, cultivar-specific, and exhibit significantly higher variance.
3.  Any reported validation metric reflects **the model's ability to fit synthetic data distributions**, not its ability to predict real-world shelf life.

### Academic Consequence

> [!CAUTION]
> A thesis examiner will ask: *"Can your model predict shelf life from a real mango photograph?"*  With the current synthetic-only training regime, the honest answer is **no**. The model has never seen a real mango image.

### Resolution

1.  **Explicitly frame synthetic data as a proof-of-concept** in the thesis. State that the pipeline architecture and training loop are validated, but real-world performance is untested.
2.  Collect even a small set (50–100) of real mango images with paired sensor readings and destructive measurements. Publicly available datasets include:
    -   [Kisan.ai Mango Dataset](https://www.kaggle.com/) (various cultivar images)
    -   Manual collection using a smartphone + DHT22 temperature/humidity sensor at a University of Ghana laboratory or local market
3.  If real data collection is infeasible, document this as a limitation and present the synthetic results under a clearly labelled "Simulated Experiment" section.

## 3. HIGH — Severely Imbalanced Target Distribution

### Severity: 🟠 High

### The Problem

Examination of `preprocessed_data.csv` reveals a severe class imbalance in the regression target:

| `days_remaining` | Row Count (Approx.) | Proportion |
| :--- | :---: | :---: |
| 0 | ~168 | ~50% |
| 1–6 | ~72 | ~21% |
| 7–20 | ~97 | ~29% |

**Half the dataset has `days_remaining = 0`** (i.e., spoiled fruit). This occurs because:
-   Ambient mangoes (max shelf life 7 days) produce `days_remaining = 0` for days 7–14 (8 out of 14 days).
-   The formula `days_remaining = max(0, max_shelf_life - day)` clamps negative values to zero.

### Academic Consequence

The model is heavily biased toward predicting `0`. Validation MAE may appear low because predicting "0 days" for half the dataset is trivially correct. A model that always predicted `0` would achieve an MAE of approximately **3.6 days** on this distribution — not far from the reported 2.22 days.

### Resolution

1.  **Report per-class metrics**: Compute MAE separately for fresh (days 1–6), transitional (days 7–14), and spoiled (day 0) samples.
2.  **Apply target-aware sampling**: Use a weighted sampler (`torch.utils.data.WeightedRandomSampler`) to oversample underrepresented days.
3.  **Remove or limit clamped samples**: Consider excluding days beyond the stated maximum shelf life (e.g., ambient samples past day 7) since they carry no predictive information.

## 4. HIGH — Missing Feature Normalisation

### Severity: 🟠 High

### The Problem

In [`tabular_branch.py` (line 82)](file:///c:/Users/julth/repo/Smart-Shelf-Life-Prediction-Model/model/src/tabular_branch.py#L82), the docstring correctly notes:

> *"normalisation should be applied upstream via StandardScaler before training"*

However, **no normalisation is applied anywhere in the pipeline**. In [`TabularFeatureEncoder.forward()` (line 128–132)](file:///c:/Users/julth/repo/Smart-Shelf-Life-Prediction-Model/model/src/tabular_branch.py#L128-L132):

```python
for col in NUMERICAL_COLS:
    values = [float(r.get(col) or 0.0) for r in records]
    numerical_parts.append(
        torch.tensor(values, dtype=torch.float32).unsqueeze(1)
    )
```

Temperature values range from ~4°C to ~32°C, while humidity ranges from ~55% to ~95%. These raw, unnormalised inputs create:
-   **Gradient scale imbalance**: Features with larger magnitudes dominate gradient updates.
-   **Slower convergence**: The Adam optimizer partially mitigates this, but normalisation would significantly improve training stability and speed.

### Resolution

Fit a `StandardScaler` on training data statistics and apply it during both training and inference:

```python
from sklearn.preprocessing import StandardScaler

scaler = StandardScaler()
train_df[NUMERICAL_COLS] = scaler.fit_transform(train_df[NUMERICAL_COLS])
val_df[NUMERICAL_COLS]   = scaler.transform(val_df[NUMERICAL_COLS])

# Save scaler for inference
import joblib
joblib.dump(scaler, 'checkpoints/tabular_scaler.pkl')
```

## 6. MEDIUM — Statistical Evaluation Gaps

### Severity: 🟡 Medium

### Issues Identified

1.  **Single-split evaluation**: Results from one 80/20 split are reported. Any variation in the split would change the metrics, making the results unreliable.
    -   **Fix**: Use 5-Fold Group Cross-Validation. Report metrics as $\text{mean} \pm \text{std}$ (e.g., $\text{MAE} = 2.22 \pm 0.35$).

2.  **No confidence intervals**: Point estimates without variance convey false precision.

3.  **R² score missing from training logs**: The [`train.py`](file:///c:/Users/julth/repo/Smart-Shelf-Life-Prediction-Model/model/src/train.py) training loop prints MAE and RMSE but not R². A separate [`evaluate_metrics.py`](file:///c:/Users/julth/repo/Smart-Shelf-Life-Prediction-Model/model/src/evaluate_metrics.py) script exists but is not integrated into the main pipeline.

4.  **No residual analysis**: Plotting `predicted vs actual` scatter plots and residual distributions would reveal systematic biases (e.g., the model overestimates shelf life for nearly-spoiled fruit).

5.  **No per-environment breakdown**: Ambient and controlled storage regimes have fundamentally different degradation curves (7 vs 21 days). Aggregated metrics mask poor performance in one regime.

### Recommended Evaluation Framework

```
For each of 5 folds:
├── Overall: MAE, RMSE, R², Adjusted R²
├── Per-Environment: Ambient MAE, Controlled MAE
├── Per-Range: Fresh (≥4 days) MAE, Transitional (1–3 days) MAE, Spoiled (0 days) MAE
├── Predicted vs Actual scatter plot
└── Residual histogram
```

## 7. MEDIUM — LUPI Implementation Concerns

### Severity: 🟡 Medium

### 7a. Shared Optimiser for Teacher and Student

In [`train.py` (lines 266–295)](file:///c:/Users/julth/repo/Smart-Shelf-Life-Prediction-Model/model/src/train.py#L266-L295), **both** the Teacher and Student parameters are placed into a single Adam optimizer:

```python
for name, param in teacher.named_parameters():
    if not param.requires_grad:
        continue
    if 'image_branch.backbone.features' in name:
        backbone_params.append(param)
    else:
        head_params.append(param)
```

This means Teacher weight updates are influenced by the Student's loss term and vice versa via the joint loss (`loss = loss_teacher + loss_student`). While this is a valid design choice, it deviates from the standard LUPI protocol where the Teacher is pre-trained separately and then frozen during Student distillation.

**Academic risk**: An examiner may ask why you chose joint optimisation over the canonical two-phase approach. Be prepared to justify this explicitly.

### 7b. Teacher Bio-Branch Is Too Shallow

The [`bio_branch`](file:///c:/Users/julth/repo/Smart-Shelf-Life-Prediction-Model/model/src/fusion_model.py#L288-L293) is a single linear layer (3 → 64) with BatchNorm, ReLU, and Dropout:

```python
self.bio_branch = nn.Sequential(
    nn.Linear(3, bio_output_dim),
    nn.BatchNorm1d(bio_output_dim),
    nn.ReLU(inplace=True),
    nn.Dropout(p=dropout_fusion * 0.5),
)
```

A single linear layer can only capture linear relationships between Brix/pH/Texture and the output. The non-linear interaction effects (e.g., low pH + high Brix indicates over-ripeness) require at least one hidden layer.

### 7c. No Quantification of Distillation Benefit

There is no comparison between:
-   Student trained **with** Teacher distillation (current LUPI setup)
-   Student trained **without** Teacher (standard supervised learning)

Without this comparison, you cannot claim that the LUPI framework adds value.

## 10. CONTEXTUAL — Local Applicability to Ghana

### Relevance to Final Year Project Context

As a University of Ghana final year project, the examiner panel will likely evaluate **local relevance and real-world applicability**. Consider addressing the following in your thesis:

### Strengths for Local Context
-   **Non-destructive inference**: Ghana's mango supply chain (from the Northern Region to Accra markets) lacks laboratory infrastructure for Brix/pH testing. A smartphone-based prediction tool is highly practical.
-   **Ambient vs. cold-storage modelling**: Many Ghanaian markets operate under ambient conditions (30–35°C). The model explicitly handles this regime.
-   **Lightweight architecture**: EfficientNet-B0 can run on mid-range Android devices common in Ghana.

### Gaps to Address
1.  **Cultivar specificity**: Ghana's dominant mango cultivars (Keitt, Kent, Palmer, and the local Daboshe) have different ripening colour profiles. Keitt, for instance, stays green even when ripe — this would confuse the image backbone entirely. Document this as a limitation and propose a `cultivar` embedding.
2.  **Market environmental conditions**: Ghanaian open-air markets have highly variable lighting (direct sunlight, shade, artificial light). The current image augmentation pipeline may not adequately simulate these conditions.
3.  **Data collection feasibility**: Propose that future work could collect real data by partnering with the University of Ghana's Department of Food Science or the CSIR-Food Research Institute.
